# React Native App Project

The React Native Project is to build a Movie Rental App that leverages the TMDB API for the movie data.

The App will have 3 screens, two dialogs, two Context API files, a global theme, a custom font, use AsyncStorage or SecureStorage, use React Navigation or Expo-Router Stack for navigation, and uses the ReactNativeElement UI Kit for some components.

# Navigation

The app has three screens:

  • A Home screen that lets the user search for movies via the TMBD API, and then displays a scrollable list of movie cards.
  • A Rented screen that displays a scrollable list of movie cards built from the list of movies that have been rented by the user.
  • A Watch screen that lets the user watch a placeholder video and mark a rented video as watched, which will remove it from the saved list of rented movies.

You can use EITHER expo-router or react navigation for this project.

The Home Screen feature list:

  • There needs to be a link in the header to go to the Rented screen.
  • There needs to be a FAB for the user to open the search dialog.
  • The search Dialog needs an Input element and a submit Button.
  • Pressing the submit Button in the search Dialog will call a function in the Search Context to make the API call.
  • Each Movie Card displayed needs to have a "rent" Button.
  • Pressing the "rent" Button displays a confirmation Dialog that will show the title of the movie, an invented rental price, a Confirm, and a Cancel Button. The Cancel Button just hides the Dialog. The Confirm Dialog will call a method in the Rented Context to add the movie object to the Storage data.
  • After a movie is rented, the card should be removed from the screen by updating the Screen Context movie data array.

The Rented Screen feature list:

  • When the user navigates to the Rented Screen, the scrollable list of Movie Cards will be built using the data from the Rented Context.
  • Each Movie Card will need to have a "watch" Button.
  • Clicking the "watch" Button will navigate to the Watch screen and pass the movie id as a param.
  • The header should have a back arrow on iOS.

The Watch Screen feature list:

  • The title of the selected movie should be displayed when the movie is not playing.
  • There needs to be a Button to mark the movie as watched, which is visible when the movie is not playing.
  • When the user rotates the device to landscape, the video should automatically enter Fullscreen mode.
  • When the movie is marked as watched, it needs to be removed from the Storage via the context object. After it has been removed, then send the user back to the Rented screen.

# Data

There needs to be two context provider files.

  1. One for making the fetch call to the API and saving the movie array data as a state variable to be used on the home screen. It should have methods for removing an item from the array, as well as, for making new fetch calls and updating the data.
  2. One for communicating with Async or Secure Storage and creating a state variable with an array of movie objects. It will need to keep the state array and the data in Storage in sync. It will also need methods for adding a movie to the rented array or removing a movie from the rented array.

Each of the Context objects will have a custom hook that lets any component access the data from either provider.

For the TMDB API you will need to register for an account on the website (opens new window). Once you are logged in you will be able to access your Authentication token. This token needs to be used each time you make an API request.

# Theming and Styles

Wrap your app in a ThemeProvider from ReactNativeElements.

Create a global theme in a single file inside your theme folder, where you define your light and dark colour schemes, plus component styles.

If you want, you can create styled components or StyleSheet objects inside the global theme file too, which can be used throughout your app.

Use the ReactNativeElements Card and Dialog component for the movie cards and the search and confirmation dialog.

# General Requirements

Organize your files into folders by functionality beyond navigation. There should be a context folder for the two data providers, a components folder for reusable components like your movie card, and a theme folder for your global theme.

All errors should be handled within your code.

Best practices for coding should always be used.

Add comments in each function outlining their purpose.

Minimum of 5 commits to your Github Repo.

Add a custom launcher Icon and Splashscreen image for your app.

Consistent style for content throughout the application.

Best practices for design and accessibility are applied throughout.

# Flutter App Project

UPDATED APRIL 9th

The final project is a Flutter App that is similar to the midterm React Native app.

The app will have three screens - Home (Search), Rented, and Watch.

The navigation between the Home|Search, Rented, and Watch screens needs to be done with Navigator Stack calling the push() and pop() methods with the MaterialPageRoutes.

# Home Screen

The AppBar for the Home screen should show the name of the App as the title and have an action button to navigate to the Rented screen.

When first loading it should display a welcome message with instructions.

There will be a FAB at the bottom which when pressed, will show an AlertDialog widget with a Search form to search for movies by keyword.

If the search returns nothing, then display a message to that effect.

When the search results are returned they should be displayed with Card widgets. They should include the movie poster image (or a placeholder when null), plus the movie title, and at least two other bits of information about the movie, like popularity and release date. Each Card should also have a button which will let the user "rent" the movie.

Clicking the "rent" button will display an AlertDialog that shows the movie title, a randomly generated price, and two buttons -> "Confirm" or "Cancel".

Clicking the confirm button needs to save the movie details of the selected movie so they can be displayed on the rented screen.

# Rented Screen

The AppBar for the Rented Screen should show a title like "Rented Movies" and have a back button which will navigate back to the Home screen.

This screen will also display a message about their being no rented movies or a list of Card widgets. They should display the same details as the cards on the home screen. The difference between these cards and the ones on the home screen, is that these cards display a "Watch" button which will use the Navigator.of(context).push() method to load the watch screen AND pass the details of the selected movie to the Watch screen.

# Watch Screen

The Watch screen will display the name of the movie being shown as the title for the AppBar and have an action button to mark the movie as watched and then go back to the Rented screen.

This screen will display one of the following:

  1. a video player that will load a video file which is a local asset loaded through pubspec.yaml.
  2. a web view that displays an embedded video from YouTube.

When the device orientation changes to landscape, make the video go fullscreen. When the device orientation changes to portrait, the video player should just fit the width of the screen.

# Data Management

There are two different data sets that need to be stored.

  1. Current search results.
  2. A list of the rented movies.

The lists of movie details can be saved through shared_preferences or in local JSON files that we read or write.

In either data persistence solution, you should use a custom Movie class, in its own file, to define the data for each object in the List. The Movie class should only have properties for the values being displayed in the Card widgets. The rest of the data, from the http search results does not need to be kept or stored.

The fetching of data should use the http package - https://pub.dev/packages/http and have its own file with a helper class.

# Styling

Best practices for design and accessibility must be followed.

There should be two ThemeData objects - a light mode and a dark mode. The device/simulator/emulator mode should be used to determine which theme is applied to all widgets in the app.

A custom font either with a local font file or GoogleFonts package is always needed.
