# Flex Layout

When building a mobile app with React Native, you will, in most cases, be using a Flexbox layout.

When adding components to your UI, remember that the default width for <View> components is auto. This means that components can collapse to the size needed for their content. They are more like HTML inline elements than block elements.

You will build the structure of a page primarily with nested <View> components. The style property flex accepts a numeric value, which is used to create a ratio of sizes between all the sibling views. Let's take a basic layout that has two <View> components inside a parent <View>. In the styles object there will be three properties: parent, a, and b.

<View style={styles.parent}>
  <View style={styles.a}>
    <Text>A</Text>
  </View>
  <View style={styles.b}>
    <Text>B</Text>
  </View>
</View>;

const styles = StyleSheet.create({
  parent: {
    flex: 1,
  },
  a: {
    flex: 1,
  },
  b: {
    flex: 1,
  },
});

With all three of these style objects having set flex: 1, it means that the parent will take 1/1 of the height. The default flexDirection of a <View> is column. The alternative is row. If the parent had flex: 0.5, it would be half the height of the screen. If the parent had flex: 2 it would be twice the height of the screen. HOWEVER, a <View> does not scroll by default. So, any content past the bottom of the screen will not be visible.

The two children <View> components have the same value so they get an equal proportion of the parent's main axis (column).

Note: A <ScrollView> will make the children automatically collapse to the height needed by their content. The flex proportions will be ignored.

The justifyContent property has a default of flex-start. So the elements will start at the top and only fill as much space as their content requires. Other values are flex-end, space-evenly, space-around, space-between and center.

The alignItems property has a default of value of stretch meaning that they will fill the width (in a column). Other values are, like CSS, flex-start, flex-end, center, and baseline.

The flexGrow and flexShrink default values are both zero. So, it is the content that decides the height of each child in a column.

You can always nest <View>s inside <View>s and you can change the flexDirection between row and column at each level.

# Styling of Components

For styling any of the components in React Native there are many css-like properties that we use to control the appearance of the different components. To apply the style properties to a component we can use an inline style object, like this:

<Text style={{ fontSize: 40 }}>Heading</Text>
1

All the properties are held inside an object. Remember numbers don't need quotation marks, but if you are adding units like percentage signs then you need quotation marks around it.

Or we can build a stylesheet object that the bottom of a file, or import and then use.

let styles = StyleSheet.create({
  someText: {
    fontSize: 40,
    paddingInline: 10,
    paddingBlock: 20,
  },
  otherStyle: {
    padding: 20,

# Safe Area Context

The core React Native project provides a <SafeAreaView /> component to help keep you application content from being obscured by sensor notches or status bars. However, it only works with iOS 11+. Fortunately, there is a community developed package that provides consistent support across all iOS, Android, and Web target deployments. It is called react-native-safe-area-context (opens new window) and Expo provides an optimized installer. You can add it to your project with the following npx command.

npx expo install react-native-safe-area-context
1

As its name implies this module uses React's Context API and, you can wrap your top App level JSX component with the <SafeAreaProvider> component imported from react-native-safe-area-context. Then wrap each of your view (screen) components with the <SafeAreaView> component imported from react-native-safe-area-context.

While the version with the Provider and the SafeAreaView does work, it has some glitchy things that happen on iOS. The official recommendation of how to use the react-native-safe-area-context package is with the useSafeAreaInsets hook.

The basic implementation of the useSafeAreaInsets would be this:

import { useSafeAreaInsets } from 'react-native-safe-area-context';

function SomeComponent() {
  const insets = useSafeAreaInsets();

  return (
    <View
      style={{
        flex: 1,
        justifyContent: 'space-between',
        alignItems: 'center',
        paddingTop: insets.top,
        paddingBottom: insets.bottom,
        paddingLeft: insets.left,
        paddingRight: insets.right,
      }}
    >
      <Text>Some Component</Text>
    </View>
  );
}

# With React Navigation Header

When using the React Navigation system, it will use, by default, the React Native Elements Header component. This Header component will rely on react-native-safe-area-context to create the proper space at the top of the header. This means that we don't need to use the inset value at the top of our page because the header is already creating the required space.

Now, let's look at the new version 7 of React Navigation. This version was initially released in summer 2024.

It is the primary navigation system for React Native apps.

On top of React Navigation, there has been built another approach to navigation that is similar to NextJS and its file based navigation system. This one is called expo router. Expo Router is built on top of React Navigation. It uses the functionality of React Navigation. We will be talking about this second approach next week.

# Navigating between Screens

Inside the pages folder create a couple copies of the HomeScreen.js file and name them CheeseScreen.js and PizzaScreen.js. Change the names of the function inside each to make the filename too. Update the value inside the <Text> component to reflect the name of the page too.

In each of the pages add these two imports:

import { useNavigation } from '@react-navigation/native';
import { Button } from '@react-navigation/elements';

The useNavigation import provides a hook method for accessing the navigation object used to navigate between the "screens".

Inside our default function we will call the useNavigation method to create the nav object and create a function that we will use for our navigation.

const nav = useNavigation();

function goto(page) {
  nav.navigate(page);
}

# Customizing the Stack

There are other ways that we can write the code for the <Stack.Screen> and there are other properties that we can add to both the <Stack.Navigator> and the <Stack.Screen>. In the <Stack.Navigator> we can add a screenOptions property, which is like the global version of the options property inside a <Stack.Screen>.

<Stack.Navigator
  initialRouteName="SomePage"
  screenOptions={{
    //these are the global version of the options in each Stack Screen
    title: 'On Every Page',
    headerBackButtonMenuEnabled: true, //show the back button
    headerBackTitle: 'BACK', //text beside back arrow
    headerBackTitleStyle: {fontSize: 14}, //fontSize for all headers
    headerStyle:{
      backgroundColor: '#f4511e', //background color for all headers
    }
    headerShadowVisible: true, //the bottom shadow or border
    headerTransparent: false, //background set to transparent
    header: ({ navigation, route, options, back }) => (<Header/>),
    //create a custom header component if you want
  }}
>
  <Stack.Screen name="SomePage" component={SomePage} options={{
    title:'Just this page',
    headerStyle: {
      backgroundColor: '#f4511e',
    },
    headerTitleStyle: {
      fontWeight: 'bold'
    },
    headerTitle: 'Instead of Screen name',
    headerRight: () =>(
      <Button onPress={()=>()}>Action</Button>
    ),
    }} />
</Stack.Navigator>

Both the options and screenOptions properties can be set as an object or a function that returns and object. Many of the individual properties inside these can also be either a String / Object or a function that returns the String / Object.

The options property of <Stack.Screen> can accept either a function or an object. These are the options for the header for this one specific page. The function version will be passed the references to the route and navigation objects.

# More Navigation Methods

The navigation object that we get from the useNavigation hook has a number of methods beyond just navigate().

The following methods can all be used to navigate to a new screen.

const nav = useNavigation();

nav.navigate('routeName'); //add screen to navigator stack (except same screen)
nav.push('routeName'); //add screen even if it is the same one
nav.goBack(); // remove the current screen from the stack, go back one screen
nav.popToTop(); // jump back to the first screen in the stack
nav.popTo('routeName'); //go back to a specific screen in the stack
1
2
3
4
5
6
7

The routeName values are meant to match the name property in a <Stack.Screen>.

The setParams method allows us to reload the current screen with a different set of params. You can think of params as name value pairs like you would have in a URLSearchParams object in the browser.

nav.setParams({
  id: 123,
  name: 'Sam',
});
1
2
3
4

These params can also be added to the navigation methods like this:

nav.navigate('routeName', { id: 321, name: 'Dean' });
1

If we have nested routes, then we can pass the nested route as the second parameter. We can add params to the nested route object.

nav.push('routeName', {
  'subrouteName',
  {id: 456, name: 'Castiel'}
})
1
2
3
4

We can also set a new title for a screen dynamically through the navigation object. We can do this through a button press function or inside the useEffect function.

nav.setOptions({ title: 'New Page Title In Header' });
1

The setOptions method can also be used to change other options that we would normally set inside the <Stack.Screen> component.

We can call any of these nav methods from inside a function that we create.

# The Route object

Every Screen that we build is a function that returns one or more components, typically a <View> or <ScrollView>.

Each screen component receives two navigator related objects as props: navigation, and route. When you create your component function, which accepts props, you can destructure to get the two properties.

export default HomeScreen({navigation, route}) => {
  //navigation and route are destructured from `props`
  //The navigation object is the one with the methods like goBack() and navigate()


  return (
    <View style={styles.screen}>
      <Text style={styles.title}>Home Screen</Text>
      <Text>{route.params.name}</Text>
    </View>
  );
}
